## 系统架构概览

系统采用 **SpringBoot 微服务+组件** 架构，主要模块包括前端界面、后端服务、文档解析、检索引擎与数据库、LLM 接口等。下图给出组件关系示意（*Mermaid 图*）：

- **前端**：可使用 Vue3 + TypeScript 构建用户界面，调用后端 API 进行问答交互。  
- **后端**：Spring Boot3 提供 REST 接口和业务逻辑，包括用户管理、会话管理、文档上传、调度等。使用 Spring AOP 监控、拦截日志。【2†L52-L60】  
- **文档解析**：调用 Apache Tika 等组件，支持多格式文档的文本及元数据提取【12†L1-L4】。解析结果可交由后端服务统一管理。  
- **Elasticsearch**：存储已解析文档的全文索引，支持关键词检索与混合检索策略【19†L49-L57】。  
- **向量数据库**：如 Milvus 或 Postgres+pgvector，存储文档分片的向量。负责向量相似度检索，可选择 Milvus（大规模、支持混合索引）或 pgvector（轻量、与关系数据结合）【5†L128-L132】【6†L72-L80】。  
- **RAG 检索编排**：使用 LangChain4j 的 Retrieval 增强生成功能，将检索到的内容与用户问题拼接成 Prompt，调用 LLM API 生成答案。系统可自定义 Prompt，使模型“只能引用所检索文档回答”。  
- **Redis 缓存**：缓存热点问题的完整答案或召回结果（例如：校历、食堂时间等），减少向 LLM 的调用频次，加快响应。也可作为会话存储和分布式锁。  
- **AI 接口**：接入 OpenAI GPT、Azure OpenAI、或国内大模型（须额外购买 API），执行问答生成。LangChain4j 提供 SpringBoot Starter，可自动配置模型【2†L52-L60】。

该架构兼顾扩展性与工程实践：各组件之间通过 REST/k8s Service 互联，可独立扩容。部署时可采用 Docker 容器，Nginx 作反向代理。根据负载，可横向扩展 Spring 服务、ES 节点和 Milvus 集群，实现**高并发**问答服务。

## 数据来源与更新策略

校园信息来源**多样**，可分为：

- **官方通知/规章**：学校官网、各部门（教务、学工、后勤）发布的 PDF、Word、HTML 公告与制度文件。
- **静态信息**：例如食堂菜单、营业时间、校历等，通常保存在静态网页或数据库中。
- **业务系统 API**：如课表、宿舍管理系统、图书馆系统等可能提供内部 API 接口，部分学校有较完善的数据中台【7†L9-L13】。
- **问答与 FAQ**：新生手册、常见问题列表、微信/公众号文章等。

**数据采集策略**：  

| 数据类型       | 示例            | 采集方式                           | 更新策略                             |
|------------|---------------|---------------------------------|------------------------------------|
| 官网页面      | 学校官网公告、新闻页面  | Web 爬虫定时抓取                      | 增量监测 (比较内容 Hash)，变更时更新向量  |
| PDF/Word 文档 | 学校规章制度文件    | 手动上传 / 教师后台上传                | 新文档即触发解析入库，老文档变更哈希检查更新  |
| 静态配置      | 食堂时间表、校历     | 手动收集或调用管理系统 API / 爬虫         | 定期验证（如每日）或监听系统事件更新          |
| API 数据      | 课程信息、宿舍状态   | 对接校内系统 API（若有）               | 调用接口，按需更新或定时同步                |
| 第三方问答     | 校园微信公众号等    | 手动整理关键 Q&A 文档                  | 人工维护列表，需及时补充新问答              |

表格：**数据来源与更新策略**。

- **手工收集**：项目初期可先**人工整理**典型文档（如学校文件夹），导入系统作为语料基线。  
- **爬虫抓取**：对于定期更新的网页（通知栏、餐厅公告、活动信息），可用定时爬虫（Spring `@Scheduled` 或 Quartz）抓取内容并保存。对爬取内容做 **内容 Hash** (如 MD5) 比对，只在检测到改动时才触发后续解析和向量更新，以避免重复计算。  
- **API/系统对接**：如果学校已有管理系统 API（如教务系统），可通过接口拉取关键信息。例如课程表、选课时间、考试公告等。通过定时拉取或消息订阅方式，获取变更后增量更新。  
- **人工维护**：对一些高频常见问题（如“如何申请免修”）可由老师维护结构化 Q&A 文档，由系统接口上传并生成向量。这样既保证正确性，又能迅速更新。  

**增量更新设计**：  

- 采用**增量向量更新**机制。当检测到某文档或数据变更后，只对**变化部分**（比如新增段落）重新生成 Chunk 的 Embedding，并用向量数据库 API 删除旧向量、插入新向量。避免全量重建带来的高成本。  
- 对于爬虫定时任务，可以设计多级流程：先抓取网页→与历史内容对比→提取新增内容→文档解析→向量化入库。新增部分的 Chunk 和老版本共同存在，在检索时可以通过元数据（如更新时间、版本号）标记。  
- 使用**哈希+时间戳**检测：每个文档/URL 保存最新内容哈希和更新时间，新内容哈希不匹配时触发更新【14†L40-L43】。同时记录更新时间，用于排序和过滤过旧内容。  

通过上述策略，可以实现知识库的**动态更新**与高时效性，保持回答内容与校园最新信息一致。

## 检索方案对比与推荐

### 向量数据库选型：Milvus vs pgvector

| 方案       | 适用场景                  | 优点                             | 缺点                                  |
|----------|------------------------|--------------------------------|-------------------------------------|
| **Milvus**    | 大规模向量检索, 企业级 RAG      | 支持**亿级**以上向量，**低延迟高并发**检索；分布式架构易于扩容；支持向量+标量过滤的混合查询【5†L128-L132】；提供多种索引(IVF, HNSW等)。 | 部署成本高（服务部署和管理复杂）；启动资源需求大；学习曲线相对陡峭。   |
| **pgvector**  | 中小规模 RAG, 现有 PostgreSQL 系统 | 与 PostgreSQL 结合自然，可直接在 SQL 中做向量和元数据混合过滤【6†L108-L116】；部署简单（Postgres 插件）；团队熟悉关系库易上手。 | 不支持集群分布式，单机水平伸缩有限；最大向量量级与检索并发受限；不支持 GPU 加速；高级功能有限。  |

表格：**向量数据库对比**。

- Milvus 专为向量检索设计，全球多家企业验证，可处理**数千万至数百亿**级向量【5†L157-L159】。它的分布式组件（Proxy、QueryNode、IndexNode 等）可横向扩展，适合大规模、长期增长场景【6†L88-L96】【5†L128-L132】。支持**无需重建索引的混合检索**(向量+关键词+元数据+重排)【5†L157-L159】。缺点是搭建和运维复杂，适合资源充足、对性能要求高的场景。  
- pgvector 是 PostgreSQL 的扩展，在已有关系库系统中无缝集成，最适合**中小规模**向量检索。它可以利用 SQL 做混合条件过滤（分类、时间戳、权限等）再做向量召回【6†L108-L116】。部署简单、与现有数据表合并方便，但不支持大规模分布式扩容。  
- **推荐**：如果校园数据量中等（几万～几十万向量）且注重开发效率，可使用 pgvector（PostgreSQL 托管成本低、二次开发简单）。若希望项目架构与前沿技术接轨，或者计划后续扩充到大规模应用，则推荐采用 Milvus（便于分布式扩容、可在性能与成本间权衡）【5†L128-L132】【6†L72-L80】。

### 搜索引擎与混合检索

- **Elasticsearch (ES)**：作为文档关键词检索引擎，可索引文档原文和元数据。ES 支持**混合搜索**（vector + keyword）管道，例如通过 [Retrieval API](https://www.elastic.co/guide/en/elasticsearch/reference/current/search-hybrid-search.html) 将关键词匹配和向量相似度结合【19†L49-L57】。Elastic 官方介绍，混合搜索先执行关键词和向量检索，然后用**融合策略**（如倒数排名融合 RRF 或线性加权）合并结果列表【19†L49-L57】【19†L54-L58】。优点是无缝存储稠密(embedding)与稀疏(BM25)数据、一次检索调用完成合并；缺点是随着文档量增加，ES 单机查询压力大。  
- **独立向量检索**：Milvus/pgvector 分别提供向量检索。可先用向量召回 Top-K，然后再用 ES 对这部分结果做关键词过滤或重排序（Rerank）。例如取向量召回的前 100 条，再做 BM25 排序。适用于需要大量向量运算的场景，但需额外步骤合并两端结果。  
- **Rerank 重排序**：在获取初步候选后，可再次调用 LLM 或机器学习模型对候选结果进行重排序。LangChain4j 中可配置 RetrievalAugmentor、Reranker 等组件，实现对召回内容的二次筛选。目前推荐先使用简单的 RRF 融合【19†L54-L58】，后续调整权重或使用自定义重排模型。  

**方案选型建议**：  
- **优先使用 ES + 向量**混合检索。ES 可直接存储向量字段（experimental），也可用 [向量检索插件](https://github.com/vectara/elasticsearch-vector-search)；或者用 Elasticsearch 作为关键词过滤层，向量计算由 Milvus 处理。在一次查询中利用 ES 的检索器（Retriever）同时跑文本查询和向量查询，并用 RRF/Fusion 合并结果【19†L49-L57】。  
- 对于**权重调整和性能**：起初可选用 RRF（倒数融合），简单无需精细调参【19†L54-L58】。如果出现特定场景需求，则考虑线性加权融合或开发自定义的混合评分策略。

## 文档解析与分块策略

**文档解析**：支持多种格式（PDF、Word、Markdown、HTML 等）。可采用 [Apache Tika](https://tika.apache.org/) 进行解析【12†L1-L4】。Tika 能自动检测文件类型，提取纯文本和元数据（作者、标题、段落结构等）。示例：Spring Boot 集成 Tika，可通过 `AutoDetectParser` 读取不同文档【11†L58-L61】。  
- **OCR**：对于扫描版文档（如历史公告图片），可先用 Tesseract 等 OCR 将图像转换为文本。注意字符识别错误需要后续清理。  
- **表格/图表**：对于复杂表格、图表等，Tika 通常只提取纯文本。可视需求，考虑将表格转为多段文本或 CSV 形式，或者手工标注重要表头信息做为元数据。  

**Chunk 切分策略**：RAG 的关键是**语义相关的分块**【14†L40-L43】。具体方法：  
- 首先以**文档结构**分层分块：例如按章节/标题切分为大块（L1-Chunks），再按段落细分为较小块（L2-Chunks），最后滑动窗口分句（L3-Chunks）进行语义化切分。这类似“先章节再段落再滑动窗口”混合策略【13†L1-L9】。  
- 确保每个 Chunk 含有**完整语义**：不要在句中硬切词；可以句号、列表符号或标题为分界。  
- 控制大小：一般保持每块长度在几百词内，不超过 LLM 的上下文窗口（按 tokens 计算）。既要避免过长导致嵌入模型无法捕捉重点，也要避免过短丢失上下文【14†L43-L46】。  
- **元数据设计**：每个 Chunk 应带上文档 ID、标题、位置索引（如第几章第几段）、更新时间等元信息。这样生成答案时可以明确引用来源【5†L128-L132】。  

示例：一份100页 PDF，先按目录标题拆分为章节，每章约分为 3-5 个段落。再将每段落内语义不变的内容保留，若过长可进一步按标点自然拆成两块。

## Embedding 模型选型与成本控制

**Embedding 模型**：常用选择包括 OpenAI 的文本嵌入（例如 *text-embedding-ada-002*），或开源模型如 sentence-transformers（多语言支持）。  
- *OpenAI 嵌入*：精度高、效果稳定。成本示例：**$0.0004/1K tokens** 左右（取决于模型）。若文档总令牌数百万，费用可观。优点是无需自建模型、易用性高。  
- *开源嵌入*：例如 [HuggingFace 的 [文档](https://huggingface.co/sentence-transformers) 模型] 或 [腾讯易智源](https://www.tensorlayer.ai/) 等，可在本地 GPU 推断。优点低成本、可完全离线；缺点质量或略逊于最先进模型、部署维护需求高。  
- **选型建议**：初期可使用 OpenAI API 快速验证（兼顾性能和开发效率）。若成本成问题，可后续迁移到开源模型或云厂商向量服务（如 Zilliz Cloud）。

**批量与增量计算**：  
- 利用批处理接口：批量对文档片段进行嵌入生成，一次请求可处理多个 chunks，减半调用次数（OpenAI 支持批量）。  
- **缓存策略**：为避免重复计算，对每个已转嵌入的 Chunk 保留Embedding结果（如存入数据库）。当模型或配置未变且文本未变时，直接使用缓存结果。LangChain4j 可结合缓存存储模块实现持久化缓存。  
- **增量 Embedding**：新内容到来时，仅对新增或更新的文本进行嵌入。利用内容 Hash 或版本号判断哪些 chunk 已变更，其他 chunk 直接复用旧 embedding 数据。  
- **成本控制**：限制最大查询长度（去除无关前后文），及时踢出无关用户。并对高频问题（如精确匹配常见问题）优先用缓存结果，避免反复触发 LLM API。【14†L74-L82】

## LLM 调用与 Prompt 设计

- **基础原则**：系统 Prompt 应明确**禁止模型胡编乱造**。可以在 Prompt 中加入提示，例如：“只根据检索到的文档内容回答问题，必要时引用文档名和条款；若无相关信息则回答“暂无相关规定”。避免模型自行编造信息。  
- **构造 Prompt**：将问题与检索到的 Top-K 相关文档片段一起拼入 Prompt，格式示例：  
  ```
  系统：你是校务知识问答助手。用户提问后，你只能根据给定的文本信息回答，必须给出出处来源。  
  用户：问：网球馆晚上还能预约吗？  
  文档：1.《体育中心场馆开放管理办法》第5条：“网球馆开放时间为8:00-22:00”。  
         2.《学生运动场使用须知》…  
  请基于以上文档给出回答，并引用来源。  
  ```  
  LangChain4j 的 `@SystemMessage`、`@AiService` 注解可用来注入系统级提示【2†L51-L60】【2†L101-L110】。  
- **引用溯源模板**：答案中应注明来源，如“根据《XXX 文件》第X条：XXX…” 或在末尾列出文档列表【5†L128-L132】。这不仅提升可信度，也是RAG系统的核心要求。  
- **多轮对话**：保留用户历史问题与回答作为聊天上下文，LangChain4j 支持对话记忆（chat memory），可在 Prompt 中加入历史内容，优化连续问答体验。需注意上下文长度限制，可使用 sliding window 策略截取最近内容。  

通过上述 Prompt 限制和对检索结果的引用，模型输出将严格限定于知识库内容，减少“幻觉”回答的可能【14†L40-L43】【19†L49-L57】。

## 工程化要点

- **Redis 缓存**：设计热点问答缓存表，如键为问题标准化文本，值为完整回答（包含引用）。对访问量高的 Q&A （如“本学期校历”、“食堂时间”）提前预热缓存。使用 Cache-Aside 模式：查库前先查 Redis，无数据时调用 RAG 流程并写回缓存。可根据访问频次、时间策略设置过期。  
- **限流与高并发**：为防止频繁调用 LLM API，使用限流算法（如漏桶、令牌桶）控制单个用户或系统总并发。Spring Cloud Gateway 或 RateLimiter 库可实现简单限流。同时，可为 LLM 调用设置超时策略，避免挂起。  
- **监控与日志**：关键操作（如文档解析、向量检索、LLM 请求）应记录日志，并监控指标（QPS、延迟、命中率、错误率）。使用 Spring Actuator 暴露健康接口，Prometheus + Grafana 可收集性能指标并报警。对 LLM 耗时可打印监控日志，方便优化。  
- **Docker 容器化部署**：后端和检索服务组件容器化，例如 SpringBoot 服务、Milvus/ES 可分别部署在容器中。通过 Docker Compose 或 Kubernetes 编排，各组件互联。保证环境一致性，便于演示和扩展。  
- **其他**：使用 Spring Boot 配置管理，环境隔离（开发/生产）。对静态资源（前端、文档存储）可使用对象存储或网络文件系统。  

以上工程措施确保系统**稳定可靠**且具备生产级特性。在简历中可突出对“分布式缓存”、“限流策略”、“容器化部署”的掌握。

## 安全与权限

- **用户与权限**：设计简易登录体系（如基于 JWT）。至少区分“教师/管理员用户”和“学生用户”。教师账号可以上传和更新文档、管理数据；学生账号或匿名用户只允许提问检索。接口层面可用 Spring Security 或过滤器控制。  
- **数据隐私**：校园知识库主要公开信息，不涉及敏感个人数据。仍需保证上传文档的来源合法，不泄露隐私。系统中尽量不处理学生个人学籍信息、成绩等隐私数据。  
- **安全传输**：部署时使用 HTTPS 加密通信，保护用户会话和 API 调用安全。对外部 LLM API 调用包含密钥，应妥善保管在后端，避免在前端泄露。

## 测试与评估指标

- **召回率 (Recall)**：评估检索效果。通过构造测试集（如预先标注问题与相关文档），计算系统召回正确文档的比例。目标召回率应尽可能高（可与人工标注对比）。  
- **准确率/精确率 (Precision)**：在生成答案时，评估回答内容的正确率。可邀请领域专家或使用已有规范问题集进行评测。  
- **延迟 (Latency)**：测量单次问答流程的总延时，包括检索和生成。目标实时响应（例如<3秒）。可记录**95% 响应时间**等。  
- **成本**：统计调用 LLM API 的次数和 token 数（可按月计费）。缓存命中率提升多少可转化为成本节省。如Redis命中率 80%，则API调用减少80%。  
- **系统负载**：并发用户量和资源使用（CPU、内存、GPU）统计。  

可编写自动化测试脚本模拟用户查询，持续记录上述指标，发现瓶颈并优化。例如监测 RRF 检索与简单 BM25 的召回差异，或对比不同 chunk 大小时的回答准确率。

## 风险分析与替代方案

- **RAG 召回不足**：如果向量检索错过部分相关文档，可增强混合检索策略（提高向量 TopK 数量+关键词过滤）。必要时调整chunk策略或使用 rerank 强化。  
- **文档更新滞后**：爬虫可能遇结构变更或反爬策略。备用方案：提供管理员后台，支持手动更新文档内容。  
- **模型幻觉**：若模型仍有“瞎编”风险，可限制回答模式。例如不允许“总结未提及信息”、或者回答前必须引用至少一条来源。  
- **性能瓶颈**：单实例向量库或 Elastic 达到饱和时，可扩展到多节点集群。备用方案：初期降低 Embedding 维度或使用快速索引（IVF）降低计算量。  
- **成本控制**：API调用成本高时，可批量更新减少调用次数，或改用开源模型降低费用。

## 暑期开发计划（6-10 周）

1. **第1-2周**：技术准备。熟悉 SpringBoot3、Redis、Docker 等。完成项目骨架搭建（SpringBoot 服务、前后端项目结构）。  
2. **第3周**：实现**文档上传与解析**。集成 Apache Tika，编写文件上传接口，提取文本并存储（MySQL）。   
3. **第4周**：实现**Chunk & 向量化**。设计分块规则，调用 Embedding 模型生成向量，保存到向量库（可先用 pgvector）。  
4. **第5周**：实现**RAG 问答流程**。编写检索服务：向量检索+LangChain4j 调用 LLM，形成答案。测试单轮问答功能，输出带来源的结果。  
5. **第6周**：集成**Elasticsearch 混合检索**。将文档同时存入 ES，添加关键词搜索层。实现 RRF 融合策略【19†L54-L58】。  
6. **第7周**：添加**Redis 缓存和多轮会话**。对高频问答结果缓存，避免重复计算。实现简单会话 ID 机制，保存上下文。  
7. **第8周**：增量更新与同步。搭建定时任务（`@Scheduled`）爬取或检查数据源，基于 Hash 做增量 Embedding 更新。完善文档更新流程。  
8. **第9周**：部署与优化。使用 Docker 容器部署各组件（SpringBoot、ES、Milvus等）。进行压力测试，优化并发、限流、监控配置。
